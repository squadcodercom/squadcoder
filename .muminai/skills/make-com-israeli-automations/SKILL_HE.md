# אוטומציות Make.com לישראל

## הוראות

### שלב 1: זיהוי דפוס התרחיש

לפני בניית תרחיש, יש למפות את התהליך העסקי לדפוס Make.com מתאים. אוטומציות עסקיות ישראליות נופלות לקטגוריות צפויות:

| תהליך עסקי | דפוס Make.com | מודולים מרכזיים | סוג טריגר |
|---|---|---|---|
| הפקת חשבוניות וסנכרון | Watch + Create | Morning (קהילתי), iCount (מובנה), Monday.com | Webhook / מתוזמן |
| דיווח מחזורי חיוב | Router + Aggregator | Morning, Google Sheets, HTTP | מתוזמן (חודשי/דו-חודשי) |
| שליחת הודעות ללקוחות | Watch + Iterator + Module | WhatsApp Business Cloud (מובנה), Monday.com | Webhook |
| ייצוא נתוני ERP | HTTP + JSON Parse + Router | Priority (קהילתי או HTTP), Google Sheets | מתוזמן |
| התראת תשלום | Webhook + Router + Create | Webhook של Cardcom/Tranzila/Grow/Bit, Slack/Email | מיידי (webhook) |
| הפקת מסמכים | Watch + Template + Email | Morning, Google Docs, Gmail | מבוסס אירוע |
| עיבוד מבוסס AI | AI Agent + Module Tools | כל מודול ככלי AI, Router | מבוסס אירוע / מתוזמן |

קריטריונים לבחירת דפוס:
- **צריך תגובה בזמן אמת?** השתמשו ב-webhooks (טריגרים מיידיים). אחרת, סריקה מתוזמנת.
- **מספר יעדים?** השתמשו במודול Router לפיצול הזרימה.
- **עיבוד רשימה?** השתמשו ב-Iterator למעבר על פריטים (למשל שורות בחשבונית).
- **צבירת נתונים?** השתמשו ב-Array Aggregator לפני הפלט הסופי.
- **צריך החלטת AI?** השתמשו ב-AI Agents של Make.com עם Module Tools (ראו שלב 8).

### שלב 2: הגדרת חיבורים לאפליקציות ישראליות

**Morning (חשבונית ירוקה)**

ל-Morning יש **מודול קהילתי** ב-Make.com, שנבנה על ידי Callbox ומופיע כ-"Morning by Callbox" בחנות של Make.com. חשוב: Make.com מציינים שהם לא מתחזקים את האינטגרציה הזו. המודול דורש **תוכנית Best ומעלה**.

הגדרת החיבור:

1. ב-Make.com, חפשו "Morning" בפלטת המודולים (לא "Green Invoice")
2. צרו חיבור דרך מפתח API וסוד מדשבורד Morning: הגדרות > API Integration
3. הדומיין של ה-API נשאר `api.greeninvoice.co.il` למרות שינוי המותג

פעולות זמינות (אין מודולי watch/trigger):

| פעולה | תיאור | פרמטרים עיקריים |
|---|---|---|
| Add Client | יצירת רשומת לקוח | `name`, `emails`, `taxId`, `address` |
| Add Document | יצירת חשבונית, קבלה, הצעת מחיר | `type`, `client`, `income`, `currency`, `lang` |
| Add Expense | רישום הוצאה | `supplier`, `amount`, `date`, `category` |
| Get All Clients | רשימת כל הלקוחות | פרמטרי דפדוף |
| Get All Documents | רשימת כל המסמכים | פרמטרי דפדוף |
| Search Clients | חיפוש לקוחות לפי קריטריונים | שם, מספר עוסק וכו' |
| Search Documents | חיפוש מסמכים לפי קריטריונים | `type`, `fromDate`, `toDate`, `status` |
| Search Expenses | חיפוש הוצאות לפי קריטריונים | טווח תאריכים, קטגוריה וכו' |
| Update Client | עדכון לקוח קיים | מזהה לקוח + שדות לעדכון |
| Delete Client | מחיקת רשומת לקוח | מזהה לקוח |
| Make an API Call | קריאת API ישירה | כל endpoint של Morning |

מכיוון שאין טריגרים, השתמשו באחת מהגישות הבאות:
- **סריקה מתוזמנת:** הריצו "Search Documents" בלוח זמנים (למשל כל 15 דקות) עם סינון למסמכים אחרונים
- **Webhook חיצוני:** המערכת שלכם שולחת קריאה ל-Custom Webhook של Make.com כשנוצר מסמך

מיפוי שדות עיקריים למסמכי Morning:

| שדה Morning | שדה Make.com | הערות |
|---|---|---|
| `type` | סוג מסמך | 10 = הצעת מחיר, 305 = חשבונית מס, 320 = חשבונית מס/קבלה, 330 = זיכוי/החזר, 400 = קבלה |
| `client.name` | שם לקוח | תומך בעברית |
| `currency` | קוד מטבע | `ILS` לשקל |
| `income[].price` | מחיר יחידה (שקלים) | **בשקלים עשרוניים** (למשל `price: 50` = 50 שקלים). לא באגורות! |
| `vatType` | טיפול מע"מ | 0 = פטור, 1 = כולל, 2 = לא כולל |
| `lang` | שפת מסמך | `he` לעברית, `en` לאנגלית |

**רפורמת החשבוניות 2026 (הורדת סף):** חשבוניות מס מעל הסף דורשות מספר הקצאה מרשות המסים. הסף יורד במהלך 2026:

| תאריך כניסה לתוקף | סף |
|--------------------|-----|
| 1 בינואר 2026 | 10,000 ש"ח |
| **1 ביוני 2026** | **5,000 ש"ח** |
| 1 בינואר 2027 | 5,000 ש"ח (מתוכנן להמשיך) |

ה-API של Morning תומך במספר ההקצאה דרך השדה `allocationNumber`. לחשבוניות מעל הסף הנוכחי, התרחיש שלכם חייב:
1. לבקש מספר הקצאה מ-API של רשות המסים לפני יצירת המסמך
2. להשתמש בזרימת ההקצאה המובנית של Morning (אם מופעלת בהגדרות החשבון)

בנו את בדיקת הסף כמשתנה ב-workflow, לא כמספר קשיח, מאחר שהסף מתוכנן לרדת שוב. חשבוניות מעל הסף ללא מספר הקצאה אינן תקפות.

**iCount**

ל-iCount יש **מודול Make.com מובנה** (נתמך רשמית). זו אפשרות משמעותית לאוטומציית הנהלת חשבונות ישראלית.

פעולות זמינות כוללות:
- יצירה וניהול הוצאות, לידים, משימות, אירועים
- ניהול מלאי
- ניהול לקוחות
- יצירת מסמכים (חשבוניות, קבלות, הצעות מחיר)

להגדרה: חפשו "iCount" בפלטת המודולים, התחברו עם פרטי ה-API של iCount.

**Monday.com**

ל-Monday.com יש מודול Make.com מובנה. שימוש נפוץ לניהול פרויקטים וחיוב.

**חשוב (מעבר ל-v2):** API v1 של Monday.com נתמך רק עד 1 במאי 2026. תרחישים חדשים חייבים להשתמש ב-v2. אם יש תרחיש קיים שעובד עם v1, תכננו מעבר מיידי. המודול המובנה של Make.com משתמש ב-v2 כברירת מחדל מ-2025.

1. השתמשו ב-"Watch Items" כטריגר (הגדירו ללוח ספציפי)
2. מפו ערכי עמודות לפי מזהה העמודה (column ID) ולא לפי הכותרת, כי הכותרות עשויות להיות בעברית ולהשתנות
3. לעמודות סטטוס, השתמשו באינדקס התווית (label index) ולא בטקסט העברי

**Priority ERP**

ל-Priority יש **מודול קהילתי** ב-Make.com. חפשו "Priority" בפלטת המודולים. לחלופין, השתמשו במודולי HTTP לשליטה מלאה.

למודול הקהילתי:
1. חפשו "Priority" במודולים של Make.com
2. התחברו עם פרטי Priority שלכם

לגישת מודול HTTP:
1. הוסיפו מודול HTTP "Make a request"
2. תבנית URL: `https://{your-priority-domain}/odata/Priority/tabula.ini/{company}/{entity}`
3. אפשרויות אימות: Basic Auth, Personal Access Token (PAT), או OAuth2 (Priority תומך בשלושתם)
4. הגדירו כותרת `Content-Type: application/json`
5. לערכי שדות בעברית, וודאו שגוף הבקשה מקודד ב-UTF-8

ישויות Priority נפוצות:

| ישות | נתיב OData | שימוש |
|---|---|---|
| `ORDERS` | `/ORDERS` | הזמנות מכירה |
| `AINVOICES` | `/AINVOICES` | חשבוניות חייבים |
| `PORDERS` | `/PORDERS` | הזמנות רכש |
| `LOGCOUNTERS` | `/LOGCOUNTERS` | ספירת מלאי |

**WhatsApp Business**

ל-Make.com יש **מודול מובנה רשמי של WhatsApp Business Cloud**. עדיף להשתמש בו במקום בגישת HTTP לצורך הגדרה פשוטה יותר וטיפול מובנה בשגיאות.

טריגרים ופעולות זמינים:
- **Watch Events** (טריגר): קבלת הודעות נכנסות, עדכוני סטטוס
- **Send a Message**: שליחת הודעות טקסט, תמונה, מסמך או מיקום
- **Send a Template Message**: שליחת תבניות הודעה מאושרות מראש (חובה ליזום שיחה)

להגדרה:
1. חברו את חשבון Meta Business שלכם ב-Make.com
2. בחרו את מספר WhatsApp Business שלכם
3. לתבניות בעברית, הגדירו שפת תבנית ל-`he`

לשימושים מתקדמים שהמודול המובנה לא מכסה, השתמשו במודול HTTP עם WhatsApp Cloud API. השתמשו בגרסת API עדכנית (בדקו את changelog של Meta במקום לקבע מספר גרסה).

**ספקי SMS ישראליים (דרך מודול HTTP)**

| ספק | נקודת קצה | שיטת אימות |
|---|---|---|
| 019 SMS | `https://019sms.co.il/api` | Bearer token: `Authorization: Bearer YOUR_TOKEN` |
| InforUMobile | `http://api.inforu.co.il/SendMessage.asmx` | שם משתמש + טוקן (שירות ASMX, פורמט XML) |
| SMS4Free | `https://www.sms4free.co.il/ApiSMS/SendSMS` | שלושה פרטים: `key`, `user`, `pass` |

עיינו ב-`references/make-israeli-modules.md` למפרטי נקודות קצה מלאים, פרטי אימות ודוגמאות payload.

### שלב 3: טיפול בנתונים בעברית

**ניתוח טקסט וטרנספורמציה**

בעיבוד טקסט עברי ב-Make.com:

- השתמשו בפונקציית `toString` לטיפול בטוח בערכי מחרוזות עבריים מתגובות API
- ל-regex על טקסט עברי, השתמשו בקלאסים של תווי Unicode: `\p{Hebrew}` מתאים לאותיות עבריות
- בשרשור עברית ואנגלית (למשל מספרי חשבונית), הציבו את החלק העברי ראשון כדי לשמר את סדר הקריאה RTL
- השתמשו ב-`trim` על שדות טקסט בעברית, כי חלק מה-API הישראליים מרפדים בתווי Unicode בלתי נראים (סימני LTR/RTL)

**פורמט מטבע שקלים**

פונקציית `formatNumber` של Make.com מטפלת ב-ILS:

| ביטוי | פלט | שימוש |
|---|---|---|
| `formatNumber(amount; 2; "."; ",")` | `1,234.56` | תצוגה סטנדרטית |
| `"₪" + formatNumber(amount; 2; "."; ",")` | `₪1,234.56` | עם סימן מטבע |

הערה: סימן השקל הוא Unicode U+20AA. אל תשתמשו ב-`NIS` כסימן בפלט ללקוחות.

**המרת תאריכים עבריים**

Make.com מאחסן תאריכים בפורמט ISO 8601. להצגה ישראלית:

- השתמשו ב-`formatDate(date; "DD/MM/YYYY")` לפורמט תאריך ישראלי (יום/חודש/שנה)
- לשמות חודשים בעברית, השתמשו בטבלת מיפוי (ל-Make.com אין פורמט חודשים בעברית מובנה):

| חודש | עברית |
|---|---|
| 1 | ינואר |
| 2 | פברואר |
| 3 | מרץ |
| 4 | אפריל |
| 5 | מאי |
| 6 | יוני |
| 7 | יולי |
| 8 | אוגוסט |
| 9 | ספטמבר |
| 10 | אוקטובר |
| 11 | נובמבר |
| 12 | דצמבר |

השתמשו ב-`formatDate(now; "M")` לקבלת מספר החודש, ואז מפו אותו לשם העברי דרך פונקציית switch או טבלת מיפוי במודול Set Variable.

### שלב 4: בניית דפוסי Router למחזורי חיוב ישראליים

עסקים ישראליים עובדים לפי מחזורי חיוב ספציפיים שונים מדפוסים אמריקאיים או אירופיים. השתמשו ב-Routers של Make.com לפיצול לוגיקה לפי מחזורים אלה.

**דיווח מע"מ דו-חודשי (דו"ח דו-חודשי)**

דוחות מע"מ מוגשים דו-חודשית עבור רוב העסקים (עסקים מתחת לסף מדווחים שנתית). תקופות המע"מ:

| תקופה | חודשים | מועד הגשה |
|---|---|---|
| 1 | ינואר-פברואר | 15 מרץ |
| 2 | מרץ-אפריל | 15 מאי |
| 3 | מאי-יוני | 15 יולי |
| 4 | יולי-אוגוסט | 15 ספטמבר |
| 5 | ספטמבר-אוקטובר | 15 נובמבר |
| 6 | נובמבר-דצמבר | 15 ינואר |

בנו Router עם 6 ענפים, כל אחד מסנן חשבוניות לתקופה הרלוונטית. אחרי ה-Router, השתמשו ב-Array Aggregator לסיכום סכומים לכל תקופה לדו"ח המע"מ.

השתמשו ב-Data Store של Make.com כדי לעקוב אחרי תקופות שעובדו ולמנוע עיבוד כפול. ראו שלב 7 לפרטי הגדרת Data Store.

**מקדמות מס דו-חודשיות (מקדמות)**

עצמאים וחברות מסוימות משלמים מקדמות מס דו-חודשיות, בהתאמה לתקופות הדיווח הדו-חודשיות של המע"מ. המקדמה נדרשת עד ה-15 עד ה-19 לחודש שאחרי כל תקופה:

| תקופה | חודשים | מועד תשלום |
|---|---|---|
| 1 | ינואר-פברואר | 15-19 מרץ |
| 2 | מרץ-אפריל | 15-19 מאי |
| 3 | מאי-יוני | 15-19 יולי |
| 4 | יולי-אוגוסט | 15-19 ספטמבר |
| 5 | ספטמבר-אוקטובר | 15-19 נובמבר |
| 6 | נובמבר-דצמבר | 15-19 ינואר |

שיעור המקדמה נקבע על ידי רשות המסים (בדרך כלל 5-15% לעסקים חדשים). שמרו את שיעור המקדמה ב-Data Store או במודול Set Variable של Make.com, כי הוא משתנה לפי עסק ומתעדכן מדי שנה.

**דיווח שנתי**

מועדי הגשת דו"ח שנתי משתנים לפי שיטת ההגשה:
- הגשה מקוונת: 30 אפריל
- הגשה דרך רואה חשבון: הארכה עד 31 מאי או מאוחר יותר (משתנה לפי שנה)

לאוטומציות שנתיות, תזמנו תרחיש לרוץ ב-1 ינואר שצובר את נתוני השנה הקודמת.

עיינו ב-`references/billing-cycle-patterns.md` להגדרות router מפורטות.

### שלב 5: תזמון עם הלוח הישראלי

**תזמון מודע שבת**

תרחישי Make.com שמתקשרים עם עסקים או לקוחות ישראליים צריכים להימנע מריצה בשבת. הגישה המומלצת היא להשתמש ב**מודול Hebcal הקהילתי** שזמין ב-Make.com (לא דורש מפתח API):

1. חפשו "Hebcal" בפלטת המודולים של Make.com (מודול קהילתי ב-`apps.make.com/hebcal-ryuwr8`)
2. השתמשו במודול Hebcal כדי לבדוק אם היום שבת או חג
3. הוסיפו מודול Filter אחרי Hebcal לעצירת ריצה בשבת/חגים

לגישה פשוטה יותר (אבל פחות מדויקת), השתמשו בהגדרות תזמון:

1. הגדירו את תזמון התרחיש לרוץ ימים ראשון עד חמישי בלבד
2. לריצות ביום שישי, הגדירו זמן ריצה אחרון ל-14:00 שעון ישראל
3. הימנעו משבת לחלוטין

בהגדרות תזמון Make.com:
- השתמשו באפשרות "Specify dates" והחריגו שבת
- הגדירו אזור זמן ל-`Asia/Jerusalem`

הערה: 14:00 ביום שישי הוא ברירת מחדל שמרנית ובטוחה. לזמני הדלקת נרות מדויקים, מודול Hebcal הוא בחירה טובה יותר כי הוא מתחשב בעונה ובמיקום.

**זיהוי חגים ישראליים**

השתמשו במודול Hebcal הקהילתי לזיהוי חגים. הוא מטפל בראש השנה, יום כיפור, סוכות, פסח, שבועות וחגים נוספים בלי צורך במפתח API או הגדרת מודול HTTP.

אם צריכים שליטה מדויקת יותר, אפשר להשתמש ב-REST API של Hebcal דרך מודול HTTP:

```
https://www.hebcal.com/hebcal?v=1&cfg=json&year=now&month=now&maj=on&geo=pos&latitude=32.0853&longitude=34.7818
```

נתחו את התגובה לתאריך היום. אם נמצא חג ראשי (`"category": "holiday"`), השתמשו במודול Filter לעצירת הריצה.

**שעות פעילות (ראשון-חמישי)**

שעות פעילות עסקיות בישראל הן בדרך כלל ראשון עד חמישי, 09:00-18:00. לאוטומציות B2B:
- תזמנו ריצות בין 09:00-17:00 שעון ישראל
- יום ראשון הוא היום העסקי הראשון בשבוע
- יום שישי הוא חצי יום (עד 13:00 בערך)

### שלב 6: טיפול ב-Webhooks משערי תשלום ישראליים

**Webhook של Cardcom**

Cardcom שולח בקשות POST ל-webhook URL שלכם אחרי אירועי תשלום:

1. צרו טריגר Custom Webhook ב-Make.com
2. הגדירו את "Notify URL" של Cardcom ל-URL של ה-webhook ב-Make.com (Cardcom API v11 תומך בהגדרת webhooks מודרנית)
3. שדות עיקריים ב-callback של Cardcom:

| שדה | תיאור | דוגמה |
|---|---|---|
| `OperationResponse` | קוד הצלחה/כישלון | `0` = הצלחה |
| `Amount` | סכום חיוב בשקלים | `150.00` |
| `CardOwnerID` | תעודת זהות של בעל הכרטיס | מספר בן 9 ספרות |
| `NumOfPayments` | מספר תשלומים | `3` |
| `Token` | טוקן כרטיס לחיובים חוזרים | |

**Webhook של Tranzila**

Tranzila משתמשת בזרימה מבוססת הפניה. ללכידת תוצאות:

1. צרו טריגר Custom Webhook
2. הגדירו את פרמטר `notify_url` של Tranzila
3. שדות עיקריים:

| שדה | תיאור |
|---|---|
| `Response` | `000` = מאושר |
| `sum` | סכום בשקלים |
| `ccno` | מספר כרטיס ממוסך |
| `myid` | תעודת זהות לקוח |
| `fpay` | סכום תשלום ראשון |
| `spay` | סכום תשלומים עוקבים |
| `npay` | מספר תשלומים **נוספים** (סה"כ תשלומים = npay + 1). אם npay=2, הלקוח משלם 3 תשלומים. |

**Tranzila API v2** מציג hosted payment fields מבוססי iframe לעמידה ב-PCI ותמיכה בתשלומי Bit.

**Webhook של Grow (מבית משולם)**

Grow היא חברת פינטק עצמאית של משולם (לא קשורה לבנק לאומי).

Grow שולחת התראות webhook כ-JSON POST. לאימות, בדקו את שדה `webhookKey` בגוף ה-JSON (לא בכותרת). ה-API של Grow משתמש ב-**multipart/form-data** לבקשות יוצאות, לא JSON.

1. רשמו את URL ה-webhook של Make.com בדשבורד של Grow
2. Grow שולחת JSON POST עם סוג אירוע ופרטי תשלום
3. אמתו את ה-webhook על ידי השוואת ערך `webhookKey` בגוף ה-JSON מול המפתח שהגדרתם

**Bit (בנק הפועלים)**

Bit היא פלטפורמת התשלומים P2P המובילה בישראל ויש לה API עסקי. ל-Bit Business webhooks:

1. הרשמו לגישת Bit Business API
2. הגדירו webhook URL בדשבורד של Bit Business
3. Bit שולח התראות על תשלומים שהושלמו

**שערים נוספים**

- **PayMe (של ישראכרט):** עיבוד תשלומים עם תמיכה בתשלומים. הגדירו webhook URL בדשבורד PayMe.
- **PayBox:** פתרון תשלום דיגיטלי עם התראות webhook.

לכל שערי התשלום, תמיד וודאו:
- קוד התגובה/סטטוס מצביע על הצלחה לפני עיבוד
- הסכום תואם את החיוב הצפוי
- לתשלומים, שמרו גם את הסכום הכולל וגם את הסכום לכל תשלום

### שלב 7: שימוש ב-Data Store למעקב תקופות מע"מ

Data Store של Make.com מספק אחסון קבוע למעקב אחרי תקופות חיוב בין ריצות תרחיש. זה מונע עיבוד כפול ונותן מעקב ביקורת.

**צרו Data Store** עם השדות הבאים:

| שדה | סוג | מטרה |
|---|---|---|
| `period_type` | Text | `vat_bimonthly`, `advance_bimonthly`, `annual` |
| `period_key` | Text | למשל `2026-P1`, `2026-P2`, `2026` |
| `status` | Text | `pending`, `processing`, `completed`, `filed` |
| `total_income` | Number | הכנסות מצטברות לתקופה |
| `total_expenses` | Number | הוצאות מצטברות לתקופה |
| `vat_payable` | Number | מע"מ מחושב לתשלום |
| `processed_at` | Date | מתי התרחיש רץ לאחרונה |
| `filed_at` | Date | מתי הדו"ח הוגש (הזנה ידנית) |

**דפוס שימוש:**

1. בתחילת כל תרחיש חיוב, השתמשו ב-"Search records" לבדוק אם התקופה הנוכחית כבר עובדה
2. אם הסטטוס `completed` או `filed`, דלגו על הריצה
3. אם הסטטוס `pending` או חסר, המשיכו עם התרחיש
4. אחרי העיבוד, עדכנו את הרשומה לסטטוס `completed` עם הסיכומים

זה חשוב במיוחד לתרחישי מע"מ, כי עיבוד כפול של אותה תקופה ייצור דוחות שגויים.

### שלב 8: AI Agents של Make.com

Make.com השיקה AI Agents באפריל 2025, ו-Visual AI Agents ב-Scenario Builder שוחררו ב-2026. זה מאפשר קבלת החלטות מבוססת AI בתוך האוטומציות העסקיות שלכם.

**יכולות עיקריות:**

- **Module Tools:** כל מודול Make.com יכול לשמש ככלי עבור ה-AI Agent. למשל, פעולת "Search Documents" של Morning יכולה להיות כלי שהסוכן משתמש בו כדי לחפש נתוני חשבוניות לפני קבלת החלטה.
- **תמיכה מולטי-מודלית:** AI Agents יכולים לעבד PDFs, תמונות ו-CSVs, שימושי לחילוץ נתונים מחשבוניות בעברית או מסמכים סרוקים.
- **Reasoning Panel:** איתור באגים ובדיקת תהליך קבלת ההחלטות של ה-AI Agent.

**שימושים עסקיים ישראליים ל-AI Agents:**

| שימוש | תפקיד AI Agent | Module Tools |
|---|---|---|
| סיווג חשבוניות | זיהוי סוג מסמך מצרופות מייל | Morning (Add Document), Gmail (Watch) |
| קטלוג הוצאות חכם | סיווג תיאורי הוצאות בעברית | Morning (Add Expense), Google Sheets |
| ניתוב לקוחות | הפניית פניות עברית למחלקה הנכונה | Monday.com (Create Item), WhatsApp (Send Message) |
| התאמת תשלומים | התאמת תשלומים לחשבוניות בין שערים | Webhook של Cardcom, Morning (Search Documents) |

להגדרת תרחיש AI Agent:
1. הוסיפו מודול AI Agent לתרחיש
2. הגדירו את ההנחיות של הסוכן (תומך בהנחיות בעברית)
3. הוסיפו Module Tools על ידי בחירת מודולים קיימים בתרחיש
4. הסוכן מחליט אילו כלים להפעיל לפי הקונטקסט

### שלב 9: חשיפת תרחישים ככלי MCP (שרת ה-MCP של Make.com)

Make.com מפעילה שרת MCP מתארח שמאפשר לסוכני AI (Claude Code, Cursor ולקוחות MCP נוספים) להפעיל את תרחישי Make שלכם ככלים, וכך אוטומציה עסקית ישראלית שכבר בניתם (יצירת חשבונית ב-Morning, סיכום מע"מ דו-חודשי) הופכת לכלי שהסוכן יכול לקרוא לו ישירות.

החיבור הוא דרך OAuth (`https://mcp.make.com`, אין מה לאחסן) או טוקן MCP (`https://<MAKE_ZONE>/mcp/u/<MCP_TOKEN>`). הרשאת הפעלת-התרחיש זמינה בכל התוכניות; הרשאת הניהול דורשת תוכנית בתשלום. כדי שתרחיש יופיע ככלי הוא חייב להיות **active**, מוגדר לתזמון **on-demand**, ועם קלטים, פלטים ותיאור מפורט מוגדרים (תיאור בעברית עובד). כלי הפעלת-תרחיש מתנתקים אחרי 25 שניות (OAuth) או 40 שניות (טוקן); ריצות ארוכות מחזירות `executionId` למשיכה עם `executions_get`.

עיינו ב-`references/make-mcp-server.md` לפרטי שיטות החיבור, בלוקי ההגדרה ל-Claude Code, טבלת ההרשאות ושימושים עסקיים ישראליים.

### דוגמה 1: סנכרון Morning ל-Monday.com

המשתמש אומר: "צור תרחיש Make.com שמוסיף פריט ב-Monday.com בכל פעם שנוצרת חשבונית מס ב-Morning"

פעולות:
1. הוסיפו פעולת "Search Documents" של Morning בתזמון כל 15 דקות, עם סינון לסוג 305 (חשבונית מס) ותאריך יצירה אחרון
2. הוסיפו פעולת "Create an Item" של Monday.com
3. מפו שדות: מספר חשבונית לעמודת שם (לפי column ID), שם לקוח לעמודת לקוח, סכום לעמודת סכום (סוג מספר, כבר בשקלים), תאריך לעמודת תאריך עם `formatDate`
4. הגדירו תזמון כל 15 דקות, ראשון-חמישי + שישי עד 14:00, אזור זמן `Asia/Jerusalem`

תוצאה: פריטי Monday.com חדשים נוצרים אוטומטית לכל חשבונית מס, עם שמות לקוח בעברית שמורים וסכומים בשקלים מפורמטים נכון.

### דוגמה 2: סיכום מע"מ דו-חודשי

המשתמש אומר: "בנה תרחיש שמייצר גיליון סיכום מע"מ בסוף כל תקופה דו-חודשית"

פעולות:
1. טריגר מתוזמן ל-1 במרץ, מאי, יולי, ספטמבר, נובמבר, ינואר
2. בדקו Data Store לסטטוס התקופה (דלגו אם כבר עובד)
3. הוסיפו "Search Documents" של Morning לשליפת כל המסמכים מהתקופה הדו-חודשית הקודמת
4. הוסיפו Iterator לעיבוד כל מסמך
5. הוסיפו Router עם ענפים להכנסות (סוג 305, חשבונית מס) וזיכויים (סוג 330, הודעת זיכוי)
6. הוסיפו Array Aggregator לכל ענף לסיכום סכומים
7. הוסיפו "Add Row" של Google Sheets לכתיבת תקופה, סה"כ הכנסות, סה"כ זיכויים, והפרש מע"מ
8. עדכנו רשומת Data Store עם סטטוס `completed` וסיכומים

תוצאה: סיכום מע"מ דו-חודשי אוטומטי שתואם את תקופות הדיווח של רשות המסים, עם Data Store שמונע עיבוד כפול.

### דוגמה 3: אישור הזמנה ב-WhatsApp בעברית

המשתמש אומר: "שלח הודעת WhatsApp בעברית כשלקוח מבצע הזמנה"

פעולות:
1. הוסיפו טריגר Custom Webhook לקבלת אירועי הזמנה
2. הוסיפו פעולת "Send a Template Message" של WhatsApp Business Cloud (מודול מובנה)
3. בחרו תבנית הודעה בעברית מאושרת מראש עם משתנים: שם לקוח, מספר הזמנה, סכום כולל בשקלים
4. פרמטו סכום עם `"₪" + formatNumber(amount; 2; "."; ",")`
5. השתמשו במודול Hebcal הקהילתי + Filter לתור הודעות בשבת למשלוח ביום ראשון בבוקר

תוצאה: לקוחות מקבלים אישורי WhatsApp בעברית עם סכומי שקלים מפורמטים נכון, בכבוד לשעות השבת.

### דוגמה 4: AI Agent לעיבוד חשבוניות

המשתמש אומר: "הגדר AI agent שקורא צרופות מייל ויוצר את סוג המסמך הנכון ב-Morning"

פעולות:
1. הוסיפו טריגר "Watch Emails" של Gmail עם סינון למיילים עם צרופות
2. הוסיפו מודול AI Agent של Make.com עם הנחיות בעברית: סווג את הצרופה כחשבונית, קבלה, הצעת מחיר, או הודעת זיכוי
3. הוסיפו "Add Document" של Morning כ-Module Tool, עם פרמטר type ממופה מסיווג הסוכן (305 = חשבונית מס, 320 = חשבונית מס/קבלה, 400 = קבלה, 10 = הצעת מחיר)
4. הוסיפו "Search Clients" של Morning כ-Module Tool כדי שהסוכן יוכל לחפש לקוחות קיימים
5. הוסיפו error handler route למסמכים שלא אפשר לסווג

תוצאה: עיבוד מסמכים מבוסס AI שמסווג חשבוניות בעברית אוטומטית ויוצר את סוג המסמך הנכון ב-Morning.

לדוגמה מעשית של חשיפת תרחיש חשבוניות Morning ככלי MCP שאפשר לקרוא לו מ-Claude Code, ראו `references/make-mcp-server.md`.

## מלכודות נפוצות

- סוכנים מניחים דיווח מע"מ חודשי (דפוס אמריקאי/אירופי). דיווח מע"מ בישראל הוא דו-חודשי עבור רוב העסקים. תמיד אשרו את תדירות הדיווח לפני בניית פילטרי תקופה.
- סכומים ב-API של Morning (חשבונית ירוקה) הם ב**שקלים עשרוניים** (למשל `price: 50` = 50 שקלים). אל תכפילו ב-100 ואל תמירו לאגורות. זה שונה מכמה API של שערי תשלום.
- מודול Morning ב-Make.com הוא **קהילתי שנבנה על ידי Callbox**, לא מתוחזק על ידי Make.com. אין בו מודולי trigger/watch, רק פעולות. תכננו סריקה מתוזמנת במקום טריגרים מבוססי אירוע.
- קודי סוג מסמך ב-Morning: 10 = הצעת מחיר, 305 = חשבונית מס, 320 = חשבונית מס/קבלה, 330 = זיכוי/החזר, 400 = קבלה. אלה לא קודי HTTP שסוכנים לפעמים מניחים.
- שדה `npay` של Tranzila מציין מספר תשלומים **נוספים**, לא סה"כ. אם מגדירים `npay=3`, הלקוח משלם 4 תשלומים (1 ראשון + 3 נוספים). תמיד השתמשו ב-`npay = סה"כ_תשלומים - 1`.
- פונקציות התאריך של Make.com משתמשות במספור יום בשבוע בסגנון אמריקאי (0 = ראשון). סוכנים מניחים לעיתים שני = 0 (ISO 8601). ב-Make.com, ראשון הוא 0 ושבת היא 6.
- סוכנים נוטים לתזמן ריצות יום שישי ב-17:00 או מאוחר יותר. שבת יכולה להיכנס כבר ב-16:00 בחורף. השתמשו ב-14:00 כזמן חתך בטוח, או עדיף, השתמשו במודול Hebcal הקהילתי לזמנים מדויקים.
- שמות עמודות בעברית ב-Monday.com צריכים להיות מופנים לפי מזהה עמודה (column ID), לא לפי כותרת התצוגה. סוכנים מנסים להשתמש בכותרת העברית ישירות, מה ששובר כשמשתמשים משנים שמות עמודות.
- פילטרים ב-Make.com עובדים דרך **ממשק ויזואלי** עם תפריטי בחירה, לא תחביר קוד. אין הבדל בין `=` ל-`==` כי בוחרים "equal to" מתפריט.
- שנת המס הישראלית היא ינואר-דצמבר (כמו שנה קלנדרית), אבל סוכנים מניחים לפעמים אפריל-מרץ (דפוס בריטי) או אוקטובר-ספטמבר (שנת כספים אמריקאית).
- **רפורמת החשבוניות 2026 משפיעה על אוטומציות, הסף יורד ב-1 ביוני 2026.** חשבוניות מס מעל הסף (10,000 ש"ח עד 31 במאי 2026, ואז 5,000 ש"ח החל מ-1 ביוני 2026) דורשות מספר הקצאה מרשות המסים. תרחישים שיוצרים חשבוניות חייבים לבדוק את הסכום ולכלול מספר הקצאה למסמכים מזכים. שמרו את הסף כמשתנה בתרחיש, לא כמספר קשיח.
- API v1 של Monday.com נתמך רק עד 1 במאי 2026. תרחישים חדשים חייבים להשתמש ב-v2. המודול המובנה של Make.com עובד עם v2 כברירת מחדל.
- תרחיש Make מופיע ככלי MCP רק כשהוא גם **active** וגם מוגדר לתזמון **on-demand**. סוכנים נוטים לקיים רק תנאי אחד. תרחיש מתוזמן או עם טריגר מיידי לא יופיע ברשימת כלי ה-MCP, לא משנה איך מוגדרים הקלטים והפלטים.
- כלי הפעלת-תרחיש ב-MCP מתנתקים אחרי 25 שניות (OAuth) או 40 שניות (טוקן). תרחיש צבירת מע"מ דו-חודשי עלול לרוץ יותר זמן, ואז הקריאה מחזירה `executionId` במקום התוצאה. משכו את התוצאה עם `executions_get` לפי אותו מזהה. אל תתייחסו ל-timeout ככישלון.

## פתרון בעיות

### שגיאה: "Morning API מחזיר 401 Unauthorized"
סיבה: אי-התאמה במפתח API/סוד, או שימוש בפרטי sandbox בייצור (או להפך)
פתרון: וודאו שאתם משתמשים בסביבה הנכונה. כתובת sandbox של Morning היא `https://sandbox.d.greeninvoice.co.il/api/v1/`, ייצור היא `https://api.greeninvoice.co.il/api/v1/`. חדשו את מפתח ה-API אם נדרש. הערה: הדומיין עדיין `greeninvoice.co.il` למרות שינוי המותג ל-Morning.

### שגיאה: "טקסט עברי מופיע משובש בפלט"
סיבה: אי-התאמת קידוד. חלק מה-API הישראליים מחזירים Windows-1255 או ISO-8859-8 במקום UTF-8.
פתרון: בדקו את כותרות תגובת ה-API עבור `charset`. אם לא UTF-8, הוסיפו מודול Text Parser אחרי מודול ה-HTTP והגדירו קידוד קלט שמתאים למקור. Morning ו-Monday.com משתמשים ב-UTF-8 באופן מובנה.

### שגיאה: "תרחיש Make.com רץ בשבת"
סיבה: אזור זמן מוגדר ל-UTC במקום Asia/Jerusalem, מה שגורם ללוח הזמנים להיות לא מיושר עם שעון ישראל.
פתרון: בהגדרות התרחיש, הגדירו אזור זמן ל-`Asia/Jerusalem`. השתמשו במודול Hebcal הקהילתי לזיהוי שבת אמין במקום בדיקות יום-בשבוע ידניות.

### שגיאה: "Webhook של Cardcom לא מופעל"
סיבה: ה-Custom Webhook של Make.com חייב להיות במצב "האזנה" (מופעל) לפני ש-Cardcom שולחת את ההתראה. כמו כן, Cardcom דורשת HTTPS.
פתרון: וודאו שהתרחיש פעיל וה-webhook במצב האזנה. העתיקו את כתובת ה-webhook אחרי הפעלתו. וודאו שהכתובת מתחילה ב-`https://`. בדקו עם עסקה קטנה קודם.

### שגיאה: "מודול Morning לא זמין ב-Make.com"
סיבה: מודול Morning דורש תוכנית Best ומעלה. זה מודול קהילתי, לא כלול בתוכניות Free או Core.
פתרון: שדרגו את תוכנית Make.com ל-Best או מעלה. לחלופין, השתמשו בפעולת "Make an API Call" מהמודול, או במודול HTTP כללי ישירות עם ה-REST API של Morning.

### שגיאה: "חשבונית נדחתה: חסר מספר הקצאה"
סיבה: חשבוניות מעל הסף של רפורמת החשבוניות דורשות מספר הקצאה מרשות המסים. הסף הוא 10,000 ש"ח עד 31 במאי 2026, ואז יורד ל-5,000 ש"ח ב-1 ביוני 2026.
פתרון: הוסיפו מודול Filter לפני יצירת מסמך שמשווה את הסכום למשתנה ב-workflow שמכיל את הסף הנוכחי (אל תקודדו את המספר ישירות, הוא מתוכנן לרדת שוב). אם הסכום עולה על הסף, בקשו מספר הקצאה קודם, ואז העבירו אותו בשדה `allocationNumber` ביצירת המסמך.

### שגיאה: "תרחיש Make לא מופיע ככלי MCP"
סיבה: התרחיש לא פעיל, לא מוגדר לתזמון on-demand, או שלחיבור ה-MCP חסרה הרשאת הפעלת-תרחיש.
פתרון: הגדירו את התרחיש לסטטוס active וגם לתזמון on-demand, שניהם נדרשים. וודאו שלחיבור ה-MCP יש את ההרשאה "Run your scenarios" (OAuth) או `mcp:use` (טוקן). אם רשימת הכלים מיושנת, חברו מחדש את לקוח ה-MCP כדי לרענן אותה.
