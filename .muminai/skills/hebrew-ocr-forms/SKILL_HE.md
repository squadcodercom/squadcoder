### שלב 5: טיפול ב-RTL וטקסט דו-כיווני

```python
import unicodedata

def normalize_bidi_text(text):
    """Normalize bidirectional text from Hebrew OCR output."""
    lines = text.split('\n')
    normalized = []
    for line in lines:
        # Strip bidi control characters
        clean = ''.join(
            c for c in line
            if unicodedata.category(c) != 'Cf'
        )
        clean = ' '.join(clean.split())
        if clean:
            normalized.append(clean)
    return '\n'.join(normalized)
```

הסירו ניקוד לפני התאמת מחרוזות מדויקת. הפונקציה `normalize_bidi_text()` מסירה רק תווים מקטגוריית Unicode מסוג `Cf` (פורמט). ניקוד עברי (סימני תנועה, U+0591 עד U+05C7) שייך לקטגוריה `Mn` (סימן לא-רווח), אז הוא שורד את הסינון הזה. אם טופס סרוק מכיל במקרה טקסט מנוקד, או שמנוע ה-OCR ממציא סימנים תועים, התאמת מחרוזת מדויקת או regex של מילת מפתח כמו `"גוש"` תיכשל בשקט מול `"גּוּשׁ"`. כדאי להסיר ניקוד במפורש לפני התאמת מילות מפתח:

```python
import re

NIKUD_RE = re.compile(r'[֑-ׇ]')

def strip_nikud(text):
    """Remove Hebrew diacritics so keyword regexes match reliably."""
    return NIKUD_RE.sub('', text)
```

נרמלו אותיות סופיות בהתאמה. בעברית יש חמש אותיות בצורה סופית: ך ם ן ף ץ. אלה אותן אותיות כמו כ מ נ פ צ אבל מקודדות כנקודות קוד נפרדות. בדרך כלל OCR שומר אותן נכון, אבל לוגיקת התאמת מילות מפתח והתאמה מקורבת עלולה להישבר: תווית שזוהתה עם צורה סופית לא תהיה שווה למונח חיפוש שנכתב עם הצורה הרגילה, ולהפך. כשמתאימים תוויות שדה, נרמלו את שני הצדדים לצורה אחת קודם:

```python
SOFIT_MAP = {'ך': 'כ', 'ם': 'מ', 'ן': 'נ', 'ף': 'פ', 'ץ': 'צ'}

def normalize_sofit(text):
    """Fold final-form letters to their medial forms for matching."""
    return ''.join(SOFIT_MAP.get(ch, ch) for ch in text)
```

הפעילו את `strip_nikud()` ואת `normalize_sofit()` רק על מפתח ההתאמה, אף פעם לא על הערך שאתם שומרים או מציגים, האותיות הסופיות והניקוד הם חלק מעברית תקינה וחייבים להישמר בפלט.

## מנועי OCR אפשריים

המנוע `Tesseract` הוא בסיס חינמי טוב לדפוס עברי, אבל ממשקי Vision מודרניים בענן משתווים אליו או עדיפים עליו בסריקות רועשות ובטפסים חתומים. תבחרו לפי השימוש:

| מנוע | דפוס עברי | כתב יד | מודל עלות | מתי להשתמש |
|------|-----------|-------|-----------|-------------|
| Tesseract (heb+eng, LSTM) | טוב לסריקות נקיות ב-300 DPI+ | חלש | חינם, ריצה מקומית | ברירת מחדל לצינורות מקומיים, לנתונים רגישים לפרטיות |
| Google Cloud Vision – Document Text Detection | טוב מאוד, עמיד לרעש | חלקי (כתב יד דמוי דפוס בלבד) | תשלום לפי בקשה | סריקות באיכות מעורבת, אצווה גדולה, טפסי PDF |
| AWS Textract (AnalyzeDocument) | לא נתמך (שפות בכתב לטיני בלבד: EN/ES/DE/IT/FR/PT) | לא לעברית | תשלום לפי עמוד | אל תשתמשו בו לטפסים בעברית; חילוץ הטקסט/הטפסים מחזיר ג'יבריש על עברית |
| Azure AI Vision – Read API | לא ברשימת השפות הרשמית ל-OCR דפוס; בדקו על המסמכים שלכם (דיווחים על ג'יבריש בעברית) | לא אמין לעברית | תשלום לפי טרנזקציה | בדקו לפני הסתמכות לעברית; חזק ללטינית ול-PDF ארגוני |
| Claude Vision (claude-sonnet-4-6 / claude-opus-4-8) | טוב מאוד, מודע להקשר | טוב (עם הנחיה בפרומפט) | לפי טוקנים | פריסות טפסים חריגות, אימות צולב של שדות, כשרוצים שהמודל גם ינתח את הנתונים |

הערה: אף אחד מהמנועים אינו אמין לכתב יד קורסיבי על טפסים כמו נסחי טאבו ישנים. לאלה תסמנו לבדיקה ידנית במקום לנסות חילוץ אוטומטי. כשקוראים ל-Google Cloud Vision לעברית, העבירו `languageHints: ['iw']` (Vision משתמש בקוד ה-ISO הישן `iw` לעברית, לא `he`); Tesseract משתמש ב-`heb`. AWS Textract ו-Azure Read אינם אמינים לעברית, העדיפו Tesseract (`heb`), Google Cloud Vision (`iw`), או Claude Vision.

## מלכודות נפוצות
- דיוק OCR בעברית יורד משמעותית בטקסט בכתב יד, במיוחד באותיות ו', ז' ו-י' שנראות דומות. תמיד תכללו ציוני ביטחון ותסמנו תווים עם ביטחון נמוך.
- טפסים ממשלתיים ישראליים סרוקים לפעמים מכילים טקסט מעורב עברית וערבית. סוכנים עלולים לזהות את החלקים הערביים כעברית או לדלג עליהם לגמרי. שתי השפות RTL אבל בטווחי Unicode שונים.
- מספרי תעודת זהות ומספרי טלפון שחולצו ב-OCR צריכים אימות עם אלגוריתמי ספרת ביקורת אחרי החילוץ, כי שגיאות OCR של ספרה בודדת נפוצות.
- הרבה טפסים ישראליים משתמשים בהדפסת מטריצת נקודות (dot-matrix) בעברית, שיש לה דיוק OCR נמוך יותר מטקסט מודפס בלייזר. סוכנים עלולים לדווח על ביטחון גבוה מדי במסמכים ממשלתיים ישנים.
- Tesseract עלול לשנות סדר של רצפי ספרות בתוך שורת RTL. מספר כמו ת.ז. בת 9 ספרות או סכום בחשבונית שמודפס בתוך משפט עברי עלול לחזור עם קבוצות הספרות מעורבבות, גם כשכל ספרה בנפרד זוהתה נכון. אל תסמכו על הסדר הגולמי משמאל לימין של מספרים שחולצו משורה מעורבת עברית/ספרות, עגנו כל מספר מחדש לתווית השדה שלו ואמתו מחדש עם כללי ספרת ביקורת או טווח.
- הסקריפטים המצורפים מקבלים רק קובץ תמונה בודד. הרבה טפסים ישראליים הם קובצי PDF רב-עמודיים (חבילת טופס 106, נסח טאבו רב-דפי). המירו כל עמוד PDF לתמונה קודם, למשל עם `pdf2image` (`convert_from_path`) או `pdftoppm`, ואז הריצו עיבוד מקדים ו-OCR לכל עמוד ומזגו את השדות שחולצו. אל תעבירו PDF גולמי ל-`preprocess_image.py` או ל-`extract_form_fields.py`.

## פתרון בעיות

### שגיאה: "חבילת שפה עברית של Tesseract לא נמצאה"
סיבה: קובץ traineddata בעברית לא מותקן
פתרון: תתקינו עם `sudo apt-get install tesseract-ocr-heb` (Ubuntu) או `brew install tesseract-lang` (macOS). אימות עם `tesseract --list-langs`.

### שגיאה: "פלט ה-OCR משובש או הפוך"
סיבה: בעיה בסדר של טקסט דו-כיווני
פתרון: תשתמשו בפונקציה `normalize_bidi_text()`. תוודאו ש-Tesseract משתמש ב-`--oem 1` (מצב LSTM). תבדקו שהתמונה לא מראתית.

### שגיאה: "דיוק נמוך בחלקים מסוימים של הטופס"
סיבה: איכות סריקה ירודה, חותמות/חתימות חופפות לטקסט, רקעים צבעוניים
פתרון: תגבירו את העיבוד המקדים - הסרת רעש חזקה יותר, חיתוך לאזורים ספציפיים בטופס, חילוץ מבוסס ROI לפריסות טפסים מוכרות.
